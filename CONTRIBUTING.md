# 贡献指南

感谢你对「圆桌论坛」的关注！本文档介绍如何参与内容贡献。

## 如何添加新一期

1. 在 `docs/episodes/` 下创建新目录，命名格式为 `ep{编号}`（如 `ep02`）。

2. 在新目录下创建以下文件：

```
docs/episodes/ep02/
├── index.md        # 本期概览（必需）
├── roles.md        # 嘉宾角色解读（可选，如果角色有变化）
├── topic-1.md      # 第一个议题
├── topic-2.md      # 第二个议题
├── ...             # 更多议题
├── lightning.md    # 快问快答
└── references.md   # 延伸阅读
```

3. 在 `docs/.vitepress/config.mts` 中注册新一期的侧边栏和导航：

```ts
// 在 nav 中添加
{ text: '第二期', link: '/episodes/ep02/' }

// 在 sidebar 中添加
'/episodes/ep02/': [
  {
    text: '第二期：你的主题标题',
    items: [
      { text: '本期概览', link: '/episodes/ep02/' },
      { text: '议题一标题', link: '/episodes/ep02/topic-1' },
      // ...
    ]
  }
]
```

## 如何添加新议题

在对应期的目录下创建 `topic-{编号}.md`，按以下模板编写：

```markdown
---
title: "议题标题"
episode: 1
type: opening
---

# 议题标题

## 议题背景

用 2-3 段简要介绍议题背景，引用真实事件或数据。

## 数据速览

| 指标 | 数据 | 来源 |
|------|------|------|
| **指标名称** | 具体数据 | 来源名称 |

:::tip 关键洞察
一句话总结核心洞察。
:::

## 嘉宾观点

### 🎯 产品经理

> 嘉宾的完整观点，使用引用块格式。
>
> 可以包含多段，用 **加粗** 标注核心论点。

### 🎨 设计师

> 观点内容...

### 💻 前端工程师

> 观点内容...

### ⚙️ 后端工程师

> 观点内容...

### 🧪 测试工程师

> 观点内容...

## 共识与分歧

**共识：**
- 共识点 1
- 共识点 2

**分歧：**
- 分歧点 1
- 分歧点 2

## 金句收录

> "金句内容" — 角色名
```

### frontmatter 中的 type 字段

| 值 | 含义 | 说明 |
|------|------|------|
| `opening` | 开场题 | 破冰亮立场 |
| `conflict` | 冲突题 | 推动碰撞 |
| `debate` | 辩论题 | 观点交锋 |
| `closing` | 收束题 | 建设性结论 |
| `lightning` | 快问快答 | 轻松环节 |

## 如何添加新角色

如果需要引入新角色（如运维工程师、数据工程师等）：

1. 在对应期的 `roles.md` 中添加角色介绍，包含：
   - 传统角色定义
   - AI 带来的变化（用表格对比"过去"和"现在"）
   - 核心挑战
   - 护城河

2. 在 `index.md` 嘉宾表格中添加新角色。

3. 在每个 `topic-*.md` 的"嘉宾观点"部分添加该角色的观点。

4. 为角色选择一个图标前缀，保持全局一致。

## 内容规范

### 写作风格

- **语言**：中文，技术名词可保留英文（如 AI、PRD、MVP、Cursor）
- **语气**：专业但不学术，像真实讨论一样自然
- **观点呈现**：每位嘉宾的观点用引用块（`>`）格式，模拟真实发言
- **加粗**：每段观点中用 `**加粗**` 标注核心论点，方便读者快速抓取
- **长度**：每位嘉宾的单个议题观点控制在 100-200 字

### 数据要求

- 每个议题应包含"数据速览"表格，至少 3 条数据
- 数据必须注明来源（公司名 + 时间）
- 优先使用公开可查证的数据源（研报、新闻、官方公告）
- 使用 `:::tip 关键洞察` 容器高亮核心发现

### 引用格式

嘉宾观点统一使用 Markdown 引用块：

```markdown
### 🎯 产品经理

> 第一段观点内容，自然口语化。
>
> 第二段展开论述，包含 **核心论点加粗**。可以引用具体案例或数据。
```

### 共识与分歧

每个议题结尾必须包含"共识与分歧"部分，用列表分别总结。这是帮助读者快速理解讨论结果的关键。

## PR 流程

1. Fork 本仓库
2. 创建功能分支：`git checkout -b feat/ep02-topic-new`
3. 编写内容，确保符合上述规范
4. 本地预览：`pnpm docs:dev`，检查页面渲染效果
5. 提交 PR，在描述中说明：
   - 新增/修改了哪些内容
   - 数据来源是否可查证
6. 等待 Review 和合并

### 分支命名

- 新增内容：`feat/描述`（如 `feat/ep02-opening`）
- 修复问题：`fix/描述`（如 `fix/ep01-data-correction`）
- 样式优化：`style/描述`（如 `style/mobile-adaptation`）

## 本地开发

```bash
# 安装依赖
pnpm install

# 启动开发服务器（支持热更新）
pnpm docs:dev

# 构建
pnpm docs:build

# 预览构建结果
pnpm docs:preview
```
